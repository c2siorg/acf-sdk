# Cryptographic subsystem

The crypto subsystem in `sidecar/internal/crypto/` protects the IPC channel between the Python SDK (PEP) and the Go sidecar (PDP). It has two responsibilities:

1. **Authentication** — prove that every frame was sent by a party holding the shared HMAC key
2. **Replay protection** — ensure a captured valid frame cannot be re-submitted

Neither responsibility involves encryption. The channel content is not confidential — the threat model is local process isolation, not network eavesdropping.

---

## Files

| File | Responsibility |
|---|---|
| `sidecar/internal/crypto/hmac.go` | HMAC-SHA256 signing and verification |
| `sidecar/internal/crypto/nonce.go` | Per-request nonce store with TTL eviction |

---

## HMAC signing (`hmac.go`)

### Key loading

The shared key is loaded from the `ACF_HMAC_KEY` environment variable at sidecar startup. It must be a hex-encoded string (e.g. 64 hex characters = 32 raw bytes).

```
ACF_HMAC_KEY=<hex-encoded key>
```

`NewSignerFromEnv()` reads the variable, hex-decodes it, and returns a `Signer`. Startup fails fast if the variable is absent, empty, or contains invalid hex — the sidecar will not accept connections without a valid key.

The same key value must be set in the Python environment before instantiating `Firewall`. Both sides derive their `Signer` independently from the same raw bytes; no key exchange happens at runtime.

### What gets signed

The HMAC does not cover the magic byte (it is a framing sentinel, not data). It covers the concatenation:

```
signed_message = version (1B) || payload_length (4B big-endian) || nonce (16B) || payload (NB)
```

This is what the Go `SignedMessage()` and Python `signed_message()` functions produce — they must be byte-for-byte identical across both implementations.

### Sign

```go
func (s *Signer) Sign(msg []byte) []byte
```

Computes `HMAC-SHA256(key, msg)` and returns the 32-byte MAC. A new `hmac.Hash` is created for each call — there is no shared mutable state.

### Verify

```go
func (s *Signer) Verify(msg, mac []byte) bool
```

Recomputes the expected MAC and compares using `hmac.Equal`. This is a constant-time comparison — it always reads both slices in full regardless of where they first differ, preventing timing oracle attacks where an attacker learns the correct MAC one byte at a time by measuring response latency.

### Frame layout recap

```
Offset  Size   Field
──────  ─────  ──────────────────────────────────────
0       1B     Magic byte (0xAC) — not signed
1       1B     Version
2       4B     Payload length (big-endian uint32)
6       16B    Nonce
22      32B    HMAC-SHA256 over bytes [1:6+16+payloadLen]
54      NB     JSON payload (RiskContext)
```

The sidecar verifies the HMAC before deserialising the JSON. A frame with a bad MAC is dropped without touching the payload.

## Replay protection (`nonce.go`)

### Why it is needed

HMAC authentication proves the frame is genuine but does not prove it is fresh. An attacker who can observe the local socket (or intercept IPC) could capture a valid frame and replay it later. Because the HMAC is still valid on the replayed frame, authentication alone would accept it.

### Nonce store

```go
type NonceStore struct {
    mu     sync.Mutex
    m      map[string]time.Time   // nonce → expiry time
    ttl    time.Duration
    stopCh chan struct{}
}
```

Each request carries a 16-byte cryptographically random nonce generated by the SDK. The store records every nonce it has accepted, keyed by the raw bytes (stored as a Go string for map efficiency). The value is the absolute expiry time: `time.Now() + TTL`.

### Seen()

```go
func (ns *NonceStore) Seen(nonce []byte) bool
```

This is the critical path operation. It does a single atomic check-and-record under the mutex:

```
Seen(nonce)
├── nonce in map AND not yet expired  →  return true   (replay — drop connection)
└── nonce absent OR expired           →  record nonce, return false   (allow)
```

The mutex ensures that two goroutines racing with the same nonce cannot both read "not seen" and both allow the frame through. The check and record are one indivisible unit.

### TTL eviction

A background goroutine sweeps the map on every TTL interval, deleting entries whose expiry has passed:

```
t = 0        nonce "N" arrives → recorded, expires at t + TTL
t = 3s       nonce "N" replayed → still active → REJECTED
t = TTL      sweep fires → "N" deleted
t = TTL+1s   nonce "N" arrives again → no entry → accepted as fresh
```

The eviction goroutine exits cleanly when `Stop()` is called (sidecar shutdown closes `stopCh`). `Stop()` is safe to call multiple times — the second call is a no-op because reading from a closed channel returns immediately.

### TTL sizing

The replay window is bounded by the TTL. A nonce cannot be replayed after it expires. Since the sidecar's enforcement latency budget is 4–8ms, even a 30-second TTL is extremely conservative — any replay attempt would have to arrive within that window to hit an active entry. The default is configurable in `config/sidecar.yaml`.

---

## Security properties

| Property | Mechanism |
|---|---|
| Authenticity | HMAC-SHA256 with shared key — forging a frame requires knowing the key |
| Integrity | HMAC covers version + length + nonce + payload — any byte flip invalidates the MAC |
| Replay prevention | Nonce store with TTL — each 16-byte random nonce is accepted exactly once |
| Timing safety | `hmac.Equal` constant-time compare — no byte-at-a-time oracle |
| Key isolation | Key copied into `Signer` at construction — caller's slice cannot mutate it |
| Fail-closed | Missing or invalid key → sidecar refuses to start |

## Key generation (operator)

Generate a 32-byte key and hex-encode it:

```bash
# Linux / macOS
openssl rand -hex 32

# Python (cross-platform)
python3 -c "import secrets; print(secrets.token_hex(32))"
```

Set the same value in both the sidecar environment and the agent environment before starting either process.
